# Beginner Play-Through Review - Units 1-9

Method: 18 reviewers (a "confused beginner" lens + a "curriculum checker" lens per unit) played through every lesson, then a skeptic adversarially re-checked each flag against the real file. 60 raw confirmations deduped to 43 findings.

**43 findings: 5 high, 25 medium, 13 low.** Spot-verified against source; not yet fixed.

## Systemic themes (fix the root, resolve many)

### 1. House-mode renderer can't show "escaped / unstyled" consequences
The stage wraps ALL exercise html in one <div class="gfx"> and styles divs by depth-agnostic rules (.gfx div, div+div). So a line placed 'outside the box', a sibling that 'escaped the wrapper', text 'outside the tag', or a mis-spelled class that 'won't take the style' all still render as normal strap lines - the promised broken visual never appears, and the fix produces zero visible change. Reword these prompts/feedback to talk about CODE POSITION and structure (what the student can read), not an on-stage detachment the house sheet cannot produce; reserve class-drives-style demos for Unit 3 where custom CSS exists.

**Touches:** 1.4 ex3, 1.5 ex4, 2.1 ex4, 2.1 ex5, 2.3 ex4, 2.4 ex2, 2.4 ex5

### 2. Phantom UI affordances the prompts promise but the app never renders
Multiple exercises point students at controls or displays that were planned but never built: an (x,y) coordinate readout, a movable crosshair on the target corner, a 'red zone' legibility slider, an opacity slider, a console 'strip' with a RUN button, a word bank inside 'fix' exercises (fix auto-replaces on a single tap), and a friendly red error overlay that 'says' a plain-English message (the engine actually echoes the raw JS error). Beginners hunt for missing UI and stall. Fix: reword to reference only what actually renders (crosshairs at center, the code line, single-tap fix), or build the affordance once and introduce it before first use. A robust engine-level fix is a plain-language error-translation layer so every 'the overlay says...' prompt becomes true and untaught tokens (null, className, TypeError, SyntaxError) never reach a beginner.

**Touches:** 3.7 ex1, 4.1 ex1/ex5, 4.2 ex2, 4.7 ex1/ex2/ex5, 6.1 ex3, 7.1 ex3, 7.3 ex1/ex4, 7.4 ex4, 7.5 ex1/ex4, 7.6 ex1, 7.9 ex2

### 3. "Press PLAY and watch" with no visible before/after
Several demos tell the student to play and observe a change that the stage cannot show: the render loads the CORRECT/fixed JS so nothing throws (no 'loud red error'); or textContent = undefined silently BLANKS a line rather than printing the word 'undefined', so the safe and unsafe versions look identical and an observant student is pushed toward the answer marked wrong. Fix: load the broken variant (inline .replace) so the fault actually fires and add a fixedRender to recover, OR reframe as a conceptual read since PLAY shows no difference; drop every claim that 'undefined' appears on air via textContent.

**Touches:** 2.2 ex3, 2.7 ex2, 7.6 ex2, 7.6 ex4, 8.5 ex3, cp8 ex3, 9.2 ex4

### 4. Untaught tokens/concepts appear before the lesson that teaches them
Code the student must read introduces syntax ahead of the one-new-thing curve: JS if/else and == (first ever conditional, in a KERNEL that demands boolean reasoning) before any control flow is taught; var one lesson early; the GSAP word 'tween' three lessons before its gloss; translateX(-100%) as a length (contradicting 5.7's 'percent = progress, not pixels'); and linear-gradient/border-radius/line-height/text-align shown unsealed. Fix: add a teaching beat (esp. if/else near 8.4 classList) BEFORE the concept is required, record tokens in the concept ledger at true introduction, and fold/seal decorative CSS as 'boilerplate, not your job here'.

**Touches:** 2.4 ex2, 6.3 ex2, 6.4 explain, 7.5 ex3, 9.2 ex4, 9.4 ex2, 9.6 ex2

### 5. Bug's title-safe home drifts to the unsafe 40/40 pin across Unit 5
Unit 4.5 drilled the safe corner (right:96px; top:54px) and the house renderer places .gfx.bug there, but Unit 5 re-pins the bug at right:40px/top:40px everywhere - silently reverting the exact value 4.5 taught students to fix. In 5.2 ex2 (safe overlay on) this makes the bug breach title-safe at scale(1), destroying the 'scaling is what breaks it out' cause-and-effect. Fix: change every Unit 5 bug pin to 96/54 (render.css, applyOnAnswer, editable code strings, inline html style pins), which also repairs 5.2 ex2's demonstration.

**Touches:** 5.2 ex1/ex2/ex4, 5.4 ex5, 5.7 ex1-3, 5.8 ex4

### 6. Engine feedback defects: false 'live change' + raw errors that clobber authored feedback
No-slot fills fall through to engine.js's default success ('the graphic followed your choice live') even though nothing re-renders; a js-slot wrong pick that yields a SyntaxError posts __gfxError, which arrives late and OVERWRITES the exercise's own explanatory feedback with jargon; and some graders fire 'delivered/complete' before the student performs the instructed Play->Continue->Stop (or 'play twice') sequence. Fix: give each no-slot fill a tailored success stating the invisible result; stop routing __gfxError to nudge() during play (downgrade to console.warn) or skip the stage reload for wrong picks that only syntax-error; and reword prompts so what's graded matches what's asked.

**Touches:** 7.3 ex3, 7.5 ex3, 7.6 ex3, 7.7 ex3, 7.8 ex3, cp7 ex2, 9.3 ex5, 9.5 ex3, 9.5 ex5
